РЕКОМЕНДАЦИИ ПО СМЕНЕ СТРУКТУРЫ ПРОЕКТА
=========================================
Дата: 2026-07-02
Контекст: Завершён рефакторинг TabPanel+FileEditor → FileTabView+FileTabSrv+FileTabModel+EditorWidget.
Планируется отделять логику от вью в остальных компонентах.

Текущая проблема: плоская core/ уже содержит 8 файлов разной природы (models, views, services,
app — всё вперемешку). При рефакторинге MsgPanel, PluginManager, HTMLEditor, EditorApp количество
файлов удвоится, и плоская структура перестанет быть навигабельной.

РЕКОМЕНДАЦИЯ 1: СУБПАКЕТЫ core/models/, core/views/, core/services/
-------------------------------------------------------------------
Перейти от плоской core/ к трём субпакетам по архитектурной роли:

core/
  __init__.py
  editor_app.py              ← QMainWindow (остаётся точкой сборки)
  models/
    __init__.py
    document.py              ← Document (уже есть)
    file_tab_model.py         ← FileTabModel (уже есть)
    message_model.py          ← новая модель сообщений для MsgPanel
    plugin_registry.py        ← новая модель: состояние всех плагинов
  views/
    __init__.py
    editor_widget.py          ← EditorWidget (уже есть)
    file_tab_view.py          ← FileTabView (уже есть)
    html_editor_widget.py     ← HTMLEditor + HtmlHighlighter
    msg_panel_view.py         ← MsgPanel как чистая view (переименовать)
  services/
    __init__.py
    file_tab_srv.py           ← FileTabSrv (уже есть)
    config_service.py         ← вынести из EditorApp: load/save config
    session_service.py        ← вынести из EditorApp: open_files restore
    zoom_service.py           ← вынести из EditorApp: zoom_in/zoom_out
    msg_srv.py                ← новая логика маршрутизации сообщений

Плюсы:
- Файл ищется по роли за 1 шаг (models/* → данные, views/* → UI, services/* → логика)
- Модули не нужно переименовывать — меняется только путь импорта
- Чёткий контракт: models/ не импортируют Qt, services/ импортируют Qt только через сигналы

Минусы:
- Много __init__.py (можно оставить пустыми)
- Импорты станут длиннее: from core.models.file_tab_model import FileTabModel

РЕКОМЕНДАЦИЯ 2: plugins_service/ — АНАЛОГИЧНОЕ РАЗДЕЛЕНИЕ
------------------------------------------------------------
Сейчас PluginManager смешивает 3 ответственности:
  1. Обнаружение и загрузка плагинов (логика)
  2. Хранение состояния (модель)
  3. Генерация UI (меню, тулбары, док-виджеты)

Предлагаемая структура:

plugins_service/
  __init__.py
  plugin_base.py              ← PluginBase (оставить как есть)
  plugin_registry.py          ← МОДЕЛЬ: словари _available, _active, _active_ui
  plugin_loader.py            ← СЕРВИС: discover, load_module, instantiate
  plugin_ui.py                ← VIEW: _setup_ui — генерация menu/toolbar/dock/status
  plugin_manager.py           ← ФАСАД: PluginManager — тонкая обёртка, делегирует трём выше
  plugin_widget.py            ← PluginWidget — переписать: вынести логику в модель

Принцип:
- PluginRegistry — чистый Python, тестируемый (как FileTabModel)
- PluginLoader — QObject с сигналами plugin_loaded, plugin_error
- PluginUI — QObject, создаёт QMenu/QToolBar/QDockWidget по данным из Registry
- PluginManager — фасад, сохраняет обратную совместимость для EditorApp и main.py

После выноса EditorApp становится тонким оркестратором:

РЕКОМЕНДАЦИЯ 7: dependency injection ВМЕСТО ПРЯМОГО СОЗДАНИЯ
---------------------------------------------------------------
Сейчас EditorApp создаёт всё сам (new file_tab_view, new file_tab_srv, ...).
При росте числа сервисов это приведёт к жёсткой связанности.

Предлагается: передавать зависимости через конструктор EditorApp, а не создавать внутри.

Было:
  self.file_tab_view = FileTabView(parent=self)
  self.file_tab_srv = FileTabSrv(self.file_tab_view, editor_class=HTMLEditor)

Стало (опционально):
  app = QApplication(sys.argv)
  config = ConfigService('config.json')
  msg_model = MessageModel()
  msg_view = MsgPanelView()
  msg_srv = MsgSrv(msg_model, msg_view)
  tab_view = FileTabView()
  tab_srv = FileTabSrv(tab_view, editor_class=HTMLEditor)
  editor = EditorApp(tab_view, tab_srv, msg_view, msg_srv, config)
  editor.show()

Этот подход:
- Упрощает тестирование (можно подставить mock)
- Делает граф зависимостей явным
- Не требует синглтонов или глобального состояния

РЕКОМЕНДАЦИЯ 8: МАРШРУТИЗАЦИЯ ОШИБОК ПО ВКЛАДКАМ MsgPanel
---------------------------------------------------------------
Три потока — три вкладки:

| Вкладка | Тип сообщений | Пример |
|---------|---------------|--------|
| Err     | неперехваченные — то, что код не ждал | segfault, исключение без try/except |
| Msg     | штатные — всё, что код ожидает и обрабатывает | ошибка загрузки плагина (PluginLoader.plugin_error), ошибка открытия файла, info/debug/warning |
| View    | сгенерированный текст, не для редактора | результаты валидации HTML, обработки текста |

Правило:
**Если ошибка поймана в try/except и обработана (код продолжает работу) — это штатная,
идёт в Msg с msg_type="error". Если ошибка НЕ поймана или проброшена (re-raise) — это неперехваченная, идёт в Err.**

Этот подход НЕ нарушает общепринятую практику: различие не в severity,
а в контроле выполнения (handled vs unhandled).
