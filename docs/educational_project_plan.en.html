📚 Educational Project Plan: ClearEditor

From "simple" to "clean" — learning software architecture through refactoring

About this plan: This document is a complete educational roadmap for the ClearEditor project. It guides students through the process of identifying architectural problems, planning a refactoring, and implementing a clean, layered architecture.

Target audience: Beginner to intermediate Python developers familiar with PyQt5.

1. Project Overview

ClearEditor is a multi-tab text editor built with Python and PyQt5. It includes HTML syntax highlighting, a plugin system, session persistence, and external file change detection.

🎯 Learning Objectives

  • Understand the Single Responsibility Principle and why it matters
  • Learn to separate concerns between GUI, business logic, and data
  • Implement a service layer for testable business logic
  • Use Dependency Injection to reduce coupling
  • Write automated tests for both logic and UI components
  • Experience a real-world refactoring process from start to finish

2. Phase 1: Understanding the Legacy Code

⚠️ Starting point: Branch legacy-architecture

📁 Current Structure

clear-editor/
├── core/              # Core application components
│   ├── editor_app.py     # Main window (everything in one class)
│   ├── tab_panel.py      # Tab management
│   ├── file_edit.py      # File editor (mixed logic)
│   └── msg_panel.py      # Message panel
├── plugins/           # Plugins (wordcount, textprocessing, iclear)
├── plugins_service/   # Plugin infrastructure
├── tests/             # Manual visual tests
├── specifications/    # Russian-language requirements
├── main.py            # Entry point
└── config.json        # Settings (sessions, plugins)

🔍 Identifying Problems

ProblemLocationWhy It's a Problem
Mixing ResponsibilitiesFileEdit contains file I/O logicViolates Single Responsibility Principle (SRP)
Tight CouplingEditorApp depends directly on all componentsHard to test and modify
Business Logic in GUIPlugins work directly with widgetsLogic cannot be reused without the UI
No Automated TestsOnly manual visual checksNo regression protection
Hard to ExtendNew features require changes to EditorAppViolates Open/Closed Principle (OCP)

  2. Find 3 additional architectural problems (not listed above)
  3. For each problem, suggest a possible solution in 1-2 sentences
  4. Estimate how many lines of code would need to change

3. Phase 2: Designing the Target Architecture

🎨 Core Principles

  • Layered Architecture — Presentation (GUI), Business Logic (Services), Data (Models)
  • Dependency Injection — components receive services via constructors
  • Single Responsibility — each class does one thing
  • Testability — business logic can be tested without PyQt

📁 Target Structure

clear-editor/
├── core/              # Presentation Layer (GUI)
│   ├── main_window.py    # Coordination only
│   ├── tab_panel.py      # Tab management only
│   ├── file_edit.py      # Text display only
│   └── msg_panel.py      # Message display only
│
├── services/          # Business Logic Layer
│   ├── file_service.py     # Open, save, external change tracking
│   ├── session_service.py  # Session persistence
│   ├── plugin_service.py   # Plugin management
│   └── settings_service.py # Config.json handling
│
├── models/            # Data Layer
│   ├── file_state.py      # File state (dataclass)
│   └── session_state.py   # Session state
│
├── plugins/           # Plugins (use services)
├── tests/             # Automated tests (pytest + pytest-qt)
├── main.py
└── config.json

🔄 Layer Interaction

│  - Emits signals about user actions                    │
└────────────────────────┬────────────────────────────────┘
                         │ Signals / Calls
                         ▼

                         │ Read / Write
                         ▼
┌─────────────────────────────────────────────────────────┐
│                      Data Layer                         │

4. Phase 3: The Refactoring Roadmap

Step 0: Preparation

  • Create branch refactoring-plan from legacy-architecture
  • Add REFACTORING_PLAN.md
  • Create branch refactored-architecture

1 day

Step 1: Create Models

  • models/file_state.py (dataclass)
  • Write tests for models

2-3 days

Step 2: Extract FileService

  • services/file_service.py
  • Write tests (without PyQt)
  • Adapt FileEdit

3-4 days

Step 3: Extract SessionService

  • services/session_service.py
  • Write tests
  • Adapt EditorApp

2-3 days

Step 4: Strengthen PluginService

  • Add validation
  • Error handling
  • Optional isolation

2-3 days

Step 5: Automate Testing

  • pytest + pytest-qt
  • Integration tests
  • GitHub Actions

3-4 days

Step 6: Final Assembly

  • Update README.md
  • Add COMPARISON.md
  • Create tags v1.0-legacy and v2.0-refactored

1-2 days

5. Student Assignments

🧪 Assignment 1: Code Analysis

Branch: legacy-architecture

  1. Find the class that violates SRP the most
  2. Propose how to split it
  3. Estimate the time needed for refactoring

🛠 Assignment 2: Mini-Refactoring

Branch: Create your own from legacy-architecture (e.g., feature/refactor-msg-panel)

  1. Extract logging logic from MsgPanel into services/logging_service.py
  2. Write tests for LoggingService
  3. Create a Pull Request into refactored-architecture

🔬 Assignment 3: Comparison

Branches: legacy-architecture vs refactored-architecture

  1. Compare Cyclomatic Complexity of key methods
  2. Compare line counts in key classes
  3. Write a report on the benefits of the new architecture

🚀 Assignment 4: Extension

Branch: from refactored-architecture

  1. Add a new service: BackupService (auto-save copies)
  2. Integrate it into the app using DI
  3. Write tests
