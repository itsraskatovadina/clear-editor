
План по выделению бизнес-логики в отдельный слой для вашего проекта.
Это один из самых эффективных способов улучшить архитектуру
-------------------------------------------------------------------

1. Анализ текущей ситуации: где сейчас находится логика?

Сейчас, судя по структуре, логика "размазана" по нескольким местам:

    В core/FileEdit: логика открытия/сохранения файла, подсветка синтаксиса, авто-дополнение тегов.
    В core/TabPanel: логика управления вкладками (создание, закрытие, переключение), которая тесно связана с FileEdit.
    В core/EditorApp: логика загрузки/сохранения сессии, координация между вкладками и плагинами, обработка внешних изменений файлов.
    В плагинах: бизнес-логика (подсчет слов, обработка текста) смешана с кодом интеграции в интерфейс (создание меню, кнопок).

2. Целевая архитектура: что мы хотим получить?

Мы хотим ввести сервисный слой — набор независимых классов (сервисов), каждый из которых отвечает за одну сквозную функцию. Главное окно (EditorApp) и его компоненты будут только координировать работу этих сервисов и обновлять интерфейс.

Новая структура каталогов:

clear-editor/

Шаг 2. Реализуйте FileService (Сервис для работы с файлами)

Этот сервис инкапсулирует всю логику взаимодействия с файловой системой. Он ничего не знает о PyQt.

файл services/file_service.py

import os
from pathlib import Path
from typing import List, Optional
from models.file_state import FileState

class FileService:
    def __init__(self):
        self._watched_files: dict[Path, float] = {}  # путь -> время последнего изменения

    def load_file(self, file_path: Path) -> FileState:
        """Читает файл и возвращает его состояние."""
        try:
            with open(file_path, 'r', encoding='utf-8') as f:
                content = f.read()
            self._watched_files[file_path] = file_path.stat().st_mtime
            return FileState(file_path=file_path, content=content, is_modified=False)
        except FileNotFoundError:
            raise FileNotFoundError(f"Файл не найден: {file_path}")
        except Exception as e:
            raise IOError(f"Ошибка чтения файла: {e}")

    def save_file(self, state: FileState) -> None:
        """Сохраняет состояние файла."""
        if not state.file_path:
            raise ValueError("Путь к файлу не указан")
        try:
            with open(state.file_path, 'w', encoding=state.encoding) as f:
                f.write(state.content)
            # Обновляем время отслеживания после сохранения
            self._watched_files[state.file_path] = state.file_path.stat().st_mtime
            state.is_modified = False
        except Exception as e:
            raise IOError(f"Ошибка сохранения файла: {e}")

    def check_external_changes(self, state: FileState) -> bool:
        """Проверяет, изменился ли файл снаружи."""
        if not state.file_path:
            return False
        current_mtime = state.file_path.stat().st_mtime
        last_mtime = self._watched_files.get(state.file_path)
        return last_mtime is not None and current_mtime != last_mtime

Шаг 3. Реализуйте SessionService (Сервис управления сессией)

Забирает логику сохранения/восстановления состояния из EditorApp.
    
файл services/session_service.py

import json
from pathlib import Path
from typing import List
from models.file_state import FileState

class SessionService:
    SETTINGS_FILE = "config.json"
    SESSION_KEY = "open_files"

    def __init__(self, settings_service): # Зависимость
        self.settings = settings_service

    def save_session(self, file_states: List[FileState]) -> None:
        """Сохраняет пути открытых файлов в настройки."""
        file_paths = [str(f.file_path) for f in file_states if f.file_path]
        self.settings.set(self.SESSION_KEY, file_paths)

    def load_session(self) -> List[Path]:
        """Загружает пути файлов из настроек и проверяет их существование."""
        paths = self.settings.get(self.SESSION_KEY, [])
        return [Path(p) for p in paths if Path(p).exists()]

from models.file_state import FileState

class FileEdit(QPlainTextEdit):
    # Новые сигналы для сервисного слоя
    file_content_changed = pyqtSignal(FileState)  # Когда пользователь изменил текст
    request_file_save = pyqtSignal(FileState)     # Запрос на сохранение (Ctrl+S)

    def __init__(self, parent=None):
        super().__init__(parent)
        self._state: Optional[FileState] = None
        self.document().contentsChanged.connect(self._on_content_changed)

    def set_state(self, state: FileState):
        """Устанавливает новое состояние для отображения."""
        self._state = state
        self.setPlainText(state.content)
        self.document().setModified(state.is_modified)

    def get_state(self) -> FileState:
        """Возвращает текущее состояние, обновляя контент."""
        if self._state:
            self._state.content = self.toPlainText()
            self._state.is_modified = self.document().isModified()
        return self._state

    def _on_content_changed(self):
        if self._state:
            self._state.content = self.toPlainText()
            self._state.is_modified = True
            self.file_content_changed.emit(self.get_state()) # Сигнал для внешнего мира

Шаг 5. Преобразуйте EditorApp в координатора (Controller)

Главное окно теперь управляет зависимостями и соединяет сигналы с сервисами. Это и есть слой контроллера.

файл core/main_window.py (изменения)

from services.file_service import FileService
from services.session_service import SessionService
from services.plugin_service import PluginService

class EditorApp(QMainWindow):
    def __init__(self):
        super().__init__()
        # 1. Инициализация сервисов
        self.file_service = FileService()
        self.session_service = SessionService(settings_service) # ...
        self.plugin_service = PluginService()

        # 2. Инициализация GUI (TabPanel, MsgPanel)
        self.tab_panel = TabPanel(self)
        self.setCentralWidget(self.tab_panel)

        # 3. Подписка на события от GUI
        self.tab_panel.tab_requested_open.connect(self._on_tab_open_request)
        self.tab_panel.current_tab_changed.connect(self._on_tab_changed)

        # 4. Загрузка сессии через сервис
        self._restore_session()

    def _on_tab_open_request(self, file_path: Path):
        """Обработчик запроса на открытие файла из GUI."""
        try:
            state = self.file_service.load_file(file_path)
            self.tab_panel.open_tab(state)  # Передаем состояние на отображение
        except Exception as e:
            self.msg_panel.show_error(f"Не удалось открыть файл: {e}")

    def _on_save_current_tab(self):
        """Обработчик сохранения из GUI."""
        current_edit = self.tab_panel.current_widget()
        if current_edit:
            state = current_edit.get_state()
            try:
                self.file_service.save_file(state)
                current_edit.set_state(state)  # Обновляем состояние, сбрасывая флаг модификации
            except Exception as e:
                self.msg_panel.show_error(f"Ошибка сохранения: {e}")

4. Ключевые преимущества такого подхода

    Тестируемость: FileService теперь можно протестировать с помощью pytest без запуска PyQt. Вы просто передаете ему пути к временным файлам и проверяете результат.
    Переиспользование: Логику из FileService можно легко использовать в других частях приложения (например, в плагинах для пакетной обработки файлов) без дублирования кода.
    Ясность: Каждый класс имеет одну четкую ответственность (Single Responsibility Principle). FileEdit занимается отображением, FileService — файлами.
    Гибкость: Если вы захотите сменить формат хранения сессии (например, с JSON на SQLite), вам нужно будет изменить только SessionService, не трогая GUI.

Эта реструктуризация — отличное упражнение для понимания принципов разделения ответственности и внедрения зависимостей. Она превращает ваш учебный проект в маленький, но правильно организованный "enterprise"-образец.

Вопрос - 
Стоит ли сохранить в учебных целях на github текущую версию и выложить кроме новой версии план рефакторинга?
Ответ - 
Я бы рекомендовал сохранить текущую версию и выложить план рефакторинга, но с умной организацией. 
В учебном проекте "сырой" код — это не баг, а фича. Сравнение "плохой" и "хорошей" архитектуры в одном репозитории —
 это лучший учебный материал, который вы можете предложить студентам.
-------------------------------------------------------------------
Почему стоит сохранить "сырую" версию

    Учебная ценность "плохого" кода
        Студенты видят реальную эволюцию проекта, а не только идеальный финал
        Можно сравнивать "до" и "после", понимая ценность каждого изменения
        "Сырой" код — это отличный материал для упражнений: "найдите 5 проблем в этом классе"

    Демонстрация процесса мышления
        План рефакторинга в сочетании с исходным кодом показывает, как архитектор мыслит
        Это ценнее, чем просто готовое решение

    Возможность для самостоятельной работы
        Студенты могут попробовать сами провести рефакторинг, сверяясь с вашим планом
        Это активное обучение вместо пассивного чтения

Вариант 1: Ветки (Branches) — рекомендую

main (или master)
  └── legacy-architecture  ← текущая версия (сохраняем как есть)
  └── refactored-architecture ← новая версия после рефакторинга
  
Как оформить:
    Текущую версию оставляете в main (или переименовываете в legacy-architecture)
    Создаете ветку refactoring-plan с файлом REFACTORING_PLAN.md
    Создаете ветку refactored-architecture с новым кодом

Преимущества:
    Можно легко сравнивать ветки через GitHub Compare
    Студенты могут переключаться между версиями и видеть эволюцию
    История коммитов в refactored-architecture покажет пошаговый процесс

git tag -a v1.0-legacy -m "Исходная архитектура до рефакторинга"

# Сравнение архитектур: "до" и "после"

| Аспект | До рефакторинга | После рефакторинга |
|--------|----------------|-------------------|
| **Разделение ответственности** | GUI + логика в одном классе | Четкое разделение на Model-View-Controller |
| **Тестируемость** | Только ручные тесты PyQt | Юнит-тесты для сервисов + интеграционные |
| **Зависимости** | Жесткие связи между классами | Внедрение зависимостей через конструкторы |
| **Расширяемость** | Новый функционал требует изменения ядра | Плагины + сервисы легко расширяются |
